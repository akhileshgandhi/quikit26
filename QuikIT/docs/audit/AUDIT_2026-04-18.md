**Depth**: manual review by AI assistant with full repo access. Not a replacement for a human security audit or paid SaaS scanner.

# §1. Security risks

## 🔴 S-1. Super admin impersonation: no IP allowlist
**Evidence**: `apps/quikit/app/api/super/impersonate/start/route.ts` — any super admin from any IP can mint an impersonation token.
**Risk**: compromised super-admin account from any jurisdiction can access all tenant data.
**Fix**: add optional `ALLOWED_SUPER_ADMIN_IPS` env list; 403 if request IP not in list. Also consider WebAuthn step-up for impersonation specifically.

## 🔴 S-2. NEXTAUTH_SECRET shared across apps
**Evidence**: design invariant — `packages/auth/index.ts` + `apps/quikscale/app/api/auth/impersonate/[token]/route.ts` require identical `NEXTAUTH_SECRET` across quikit and quikscale for JWT signing to work cross-app.
**Risk**: compromise of any app's environment compromises all three. Rotation requires simultaneous deploy.
**Fix**: consider per-app secrets with a shared signing key only for impersonation JWTs. Medium complexity — defer unless enterprise asks.

## 🟠 S-3. Impersonation token is a DB-stored opaque string
**Evidence**: `Impersonation.token` — random 32-byte base64url, single-use, DB-validated.
**Strength**: good — cryptographically unguessable, single-use, self-expiring.
**Weakness**: if DB is compromised, ALL currently-valid tokens leak. Also: not hashed at rest.
**Fix**: hash the token with bcrypt or SHA-256 before storing (like OAuth refresh tokens). Compare on accept. Low complexity.

## 🟠 S-4. Audit log oldValues/newValues stored as raw JSON strings
**Evidence**: `AuditLog.oldValues: String @db.Text` + same for newValues.
**Risk**: if a column later contains PII (e.g. logging the full user object), the audit log becomes a GDPR-problematic uncontrolled data sink.
**Fix**: enforce a `scrubForAudit()` helper on write; redact known PII fields (`password`, `passwordHash`, `email`, `billingEmail`) before serializing.

## 🟠 S-5. Super admin bulk-user endpoint has no row-count cap
**Evidence**: `apps/quikit/app/api/super/users/bulk/route.ts` — accepts `userIds: string[]` with no upper bound. A crafted request with 100k IDs could exhaust DB connections on a pooler.
**Fix**: cap at e.g. 100 in the Zod schema. Same check for `apps/bulk` and `orgs/bulk`.

## 🟠 S-6. Broadcast target lists leak tenant IDs between announcement admins
**Evidence**: `GET /api/super/broadcasts` returns full `targetTenantIds`, which leaks the existence and IDs of all tenants on the platform to any super admin. That's fine today (we trust super admins fully), but no partitioning exists if you later add "regional super admins" or similar scoped roles.
**Fix**: defer until multi-tier super-admin roles exist.

## 🟡 S-7. Prisma queries use string interpolation nowhere — good
**Evidence**: grep for `$queryRaw` across repo. **No findings**. Everything goes through the Prisma query builder.
**Recommendation**: keep this invariant. Block `$queryRaw` via ESLint rule for extra confidence.

## 🟡 S-8. Rate limiting coverage is inconsistent
**Evidence**: only impersonate/start, a few tenant mutation routes (see `P0-3` plan doc), and OAuth token endpoint have rate limits. Most super-admin mutations (plan edit, invoice mark-paid, app access toggle) do not.
**Risk**: compromised super admin could spam expensive operations until manual intervention.
**Fix**: add `rateLimitAsync` with liberal caps (e.g. 60/min/super-admin) in `withSuperAdminAuth`.

## 🟡 S-9. CSRF — relying on SameSite=Lax cookies only
**Evidence**: NextAuth default. No explicit CSRF tokens on mutation routes.
**Assessment**: SameSite=Lax blocks cross-site POST from third-party sites, which is the main CSRF vector. Fine for internal-tool super admin. Would fail strict banking/healthcare compliance.
**Fix**: if SOC 2 Type II is a goal, add `next-csrf` or `csurf`-style tokens for super-admin mutations.

## 🟡 S-10. User-uploaded content path: nothing on the platform yet
**Evidence**: no file upload endpoints found. `logoUrl`, `avatar`, `iconUrl` all accept URLs, not uploads.
**Risk**: DoS via 10MB image URLs; SSRF if the server ever fetches these URLs server-side.
**Fix**: when adding uploads, use signed URLs direct to S3/R2 + never fetch user-provided URLs server-side without allowlist.

## 🟢 S-11. Session cookies look correct
**Evidence**: `sessionCookieName()` in impersonate accept route correctly uses `__Secure-` prefix in prod. `httpOnly: true`, `sameSite: "lax"`, `secure: production`.

## 🟢 S-12. OAuth token storage
**Evidence**: `OAuthClient.clientSecret` stored bcrypt-hashed. Access tokens not persisted (JWT strategy). Refresh tokens in `OAuthRefreshToken` table.

# §2. Data leakage review

## 🔴 L-1. `/api/super/orgs` without filters exposes every tenant ID
**Evidence**: any authenticated super admin gets the full tenant directory. Today this is by design.
**Risk**: if a lower-privilege "read-only super admin" role is added later, this becomes a leak.
**Fix**: defer until multi-tier SA roles exist (see S-6).

## 🟠 L-2. Audit log newValues may contain tokens
**Evidence**: `Impersonation.token` is a random secret. If an audit write ever serializes the full `Impersonation` row (not just metadata), the token leaks into the audit log.
**Today**: looking at `impersonate/start/route.ts` line ~110, `newValues` is curated — token is NOT included. Safe.
**Fix**: lint rule or test to prevent this regression. Also: the `writeAuditLog` helper should scrub known-secret field names (see S-4).

## 🟠 L-3. Error messages sometimes echo user input
**Evidence**: `apps/quikit/app/api/super/plans/route.ts:76` — validation error message includes the submitted slug. Not sensitive today, but same pattern in `invoices/[tenantId]/route.ts:88` (`Tenant's plan "{slug}" is not defined`) could leak internal state.
**Fix**: sanitize — show "plan not found" without the submitted value.

## 🟡 L-4. API response shape differs when empty
**Evidence**: some endpoints return `{ success: true, data: [] }` vs `{ success: true, data: null }` inconsistently.
**Leak**: subtle fingerprinting signal for scraped APIs.
**Fix**: standardize on `{ data: [] }` for lists, `{ data: null }` for single-object-not-found.

## 🟡 L-5. OAuth `userinfo` endpoint returns full user object
**Evidence**: `apps/quikit/app/api/auth/userinfo/route.ts` — returns firstName, lastName, email, tenantId, role, isSuperAdmin.
**Risk**: `isSuperAdmin` leaks platform-level authority to apps that shouldn't need it.
**Fix**: scope claims — `isSuperAdmin` only returned to launcher itself, not to quikscale/admin.

## 🟢 L-6. KPI/Priority weekly logs are tenant-scoped
**Evidence**: every list endpoint filters by `tenantId` derived from session. Verified via grep — `findMany({ where: { tenantId } })` is the universal pattern.

# §3. API quality — per-endpoint findings

## Across-the-board observations

### 🟠 A-1. ~38 routes still using raw `requireAdmin()` / `requireSuperAdmin()` instead of wrappers
**Impact**: these routes don't log to `ApiCall`, so the analytics dashboard has huge blind spots.
**Count**: 11 admin routes + 25 super-admin routes = 36 files.
**Fix**: mechanical migration to `withAdminAuth` / `withSuperAdminAuth`. Tech-debt item #5 (still open).

### 🟠 A-2. Pagination is inconsistent
**Observation**: some list routes use shared `paginationToSkipTake`, others hand-roll `skip: (page - 1) * limit`. Page size defaults vary: 20, 25, 50, 100.
**Fix**: enforce single helper + default page size of 25 everywhere via schema.

### 🟡 A-3. Many routes do N+1 queries instead of Prisma `include`
**Example**: `apps/quikit/app/api/super/organizations/[id]/route.ts` fetches org, then members, then their userTeams separately. Could be one `include`.
**Fix**: audit per route. Low priority unless latency shows up.

### 🟡 A-4. No global error-shape middleware
**Observation**: every route has its own `try { ... } catch (error: unknown) { ... }` block. ~80% identical.
**Fix**: `withTenantAuth` already does this for its wrapped routes. Migrating per A-1 fixes this too.

## Specific routes

### 🟠 A-5. `/api/super/analytics/overview` is an 11-query dashboard
**Evidence**: `apps/quikit/app/api/super/analytics/overview/route.ts` — 11 `Promise.all` queries on every request. No caching.
**Impact**: dashboard takes ~500-1500ms; each refresh hits 11 queries.
**Fix**: Redis-cache the entire payload for 60s (`super:analytics:overview`). Invalidate on relevant writes or just let TTL expire.

### 🟠 A-6. `/api/super/tenant-health/[tenantId]` does 10 queries
**Evidence**: same file style as A-5.
**Fix**: cache for 60s per tenant.

### 🟡 A-7. `/api/broadcasts/active` does 2 sequential queries
**Evidence**: fetches all active, then fetches dismissals for this user. Could be one query with `not: { dismissals: { some: { userId } } }`.
**Fix**: low priority.

### 🟡 A-8. `/api/super/users?tenantId=X` — no index on the new filter
**Evidence**: filter uses `memberships: { some: { tenantId } }` but `Membership.tenantId` already has an index. Check query plan to confirm Prisma uses the index efficiently.

### 🟢 A-9. Feature-flag API `/api/feature-flags/me`
**Evidence**: extremely fast — single query through `getDisabledModules` which is deduped via `React.cache` during a render.

# §5. Latency / API call reduction opportunities

## Where to reduce API calls

1. **ImpersonationBanner polls `/api/auth/session` every 60s** (`packages/ui/components/impersonation-banner.tsx:44`).
   - **Fix**: switch to NextAuth's `useSession().update` pattern. ~4 req/min/active session saved.
2. **BroadcastBanner polls every 5 minutes** (`broadcast-banner.tsx:55`).
   - **Fix**: Server-Sent Events (SSE) endpoint. Only push when a new broadcast fires. Works for ~100 concurrent users, trivial to implement.
3. **Every QuikScale dashboard page hits `/api/feature-flags/me`** on mount.
   - **Fix**: already cached module-level in `useFeatureFlagsForApp` — good. Confirmed no duplicate requests.
4. **Tenant detail page fires 4 panel APIs in parallel** on mount (health, app-access, billing, analytics).
   - **Fix**: a single `/api/super/tenants/[id]/full` that returns everything in one query. Saves 3 HTTP roundtrips per page load.
5. **Analytics overview doesn't cache at all** (A-5 above).
   - **Fix**: 60s Redis cache. The data is eventual anyway.

## Where latency dominates

1. **Prisma cold start on serverless**: first request to a function hits ~800ms connection overhead.
   - **Fix**: already using connection pooler (Neon pgbouncer URL in `DATABASE_URL`). Confirm `connection_limit=1&pool_timeout=10` is set. If not, add.
2. **React.cache() usage** in `getDisabledModules` and `isTenantAppBlocked` — good. Dedupes within a render.
3. **Raw ApiCall writes are fire-and-forget** — already non-blocking. Good.
4. **`withTenantAuth` wraps every route** — adds ~2-5ms overhead (session read + 1 DB query for membership check). Acceptable.
5. **JWT decoding on every request** — <1ms. Negligible.

## How to reduce latency

- **Edge middleware** for auth checks: NextAuth's middleware runs on Edge — moving `requireAdmin` there saves a cold-start cycle. Complexity: medium.
- **Prisma Accelerate** (paid) caches queries at the edge. ~50-200ms win per cached query. Free for low volume.
- **React Server Components (RSC)** for super-admin pages: currently all pages are `"use client"` with `useEffect` fetches. RSC would skip the HTTP hop entirely. Complexity: high (rewrite).
- **HTTP caching headers** already in place on some routes (e.g. `/api/apps/switcher` has `Cache-Control: private, max-age=30, stale-while-revalidate=60`). Extend to more read-only endpoints.

# §6. Engineering overkills (things we probably over-built)

1. **`@quikit/shared/withApiLogging`** — we built a generic HOF wrapper for logging. We ended up creating app-specific wrappers anyway (`withAdminAuth`, `withSuperAdminAuth`). The generic one is dead code — can remove.
2. **FF-1 cascade rule** — dot-notation hierarchy with cascade semantics is elegant but no customer has asked for "disable all kpi.* at once". A flat list with a manual "disable all children" button would be simpler.
3. **`BroadcastDismissal` per-user tracking** — for an internal platform with ~10 concurrent users, we could just use localStorage. Dismissals don't need durability across devices.
4. **Alerts engine `PlatformAlert` model with dedupe + resolve logic** — could be a simple query-based "what's on fire right now" derived view. No need to persist alert rows.
5. **`ApiCallHourlyRollup` + cron** — for <10 tenants, the raw ApiCall table has <10k rows/day. We could query raw data for analytics and skip the rollup until 100+ tenants.
6. **SlidePanel + Modal components** — two variants for the same "overlay form" job. Pick one.
7. **React.cache fallback shim** — once React 19 is universally adopted, this fallback becomes dead code. Reminder to remove.
8. **Impersonation `targetAppSlug` + reason** — stored but currently the UI only uses them for display. If unused after 6 months, trim columns.

**Principle**: these were all defensible at build time, but each is a carrying cost. Review annually.

# §9. Loopholes / edge cases discovered

1. **User can sign in with tenantId=null** — if OAuth exchange returns no `tenant_id`, NextAuth session has no `tenantId`. Many routes then 403 but some may crash on `tenantId!`. Spot-check needed.
2. **Impersonation on a suspended tenant** — what happens if a super admin tries to impersonate into a tenant whose `status = "suspended"`? Code doesn't check. Likely allows it, which is technically fine (super admin can still triage) but should be explicit.
3. **`TenantAppAccess.enabled = false` doesn't kick existing sessions** — user with an active session continues accessing the blocked app until their JWT expires (up to 7 days). Next request would 403, but server components that load data before rendering would succeed once.
4. **Feature flag cascade rule can hide unrelated routes** — if an admin disables `"opsp"`, all `opsp.*` sub-routes are gated. But `opsp` is both a module key AND a path prefix. If someone renames `/opsp` to `/okr-strategy`, the module key should stay. Keep these decoupled.
5. **`Invoice.planSlug` is snapshotted** but `Invoice.amountCents` is copied from `Plan.priceMonthly` at generation. If the plan price changes mid-billing-cycle, the invoice has the old price. By design (good) but not documented.
6. **Broadcast `endsAt` can be in the past at creation** — UI validates, server also validates. But a cron-based cleanup of ended broadcasts doesn't exist; table grows forever.
7. **`AppModuleFlag.enabled = true` rows are never created** — sparse storage assumes "no row = enabled". If someone accidentally inserts `enabled: true`, it's semantically equivalent but confusing. Defensive: `isModuleEnabled` treats presence-with-enabled=true the same as absence.
